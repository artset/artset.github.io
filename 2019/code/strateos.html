UI/UX: Compound/Inventory Linking

Due to a partnership with Eli Lily and a new lab in the works, the Strateos product was expanding to support automating compound synthesis. There had already existed an inventory section to keep track of assays. Given the new feature of compound synthesis, we needed a way to keep track of which compounds were in which containers. This is where compound/inventory linking comes in!

I conducted in depth interviews with application scientists, chemists, and internal Strateos employees in the new labs that would interact with this feature.

I identified two users that would require slightly different experiences:
Internal users: Chemists and operations at the new lab would need to create this link. The link would tend to come in the form of a CSV and handle hundreds of vials. Thus, manual support would be too laborious.
External users: Chemists at companies such as Eli Lily and Amgen work at a much smaller scale to create these links, thus manual linking would be sufficient.

Next, I did research to find out where in the existing web app would users go to to create this link -- compound or inventory?

"Chemists are always thinking about the compound, so they would go to the compound page." (User findings)

Based on the mentality of a chemist, I realized that the feature must start at the compound page. I identified three contexts in which users would interact with this feature, then mapped out a user diagram where my mocks would be based off of.

Amino Documentation Website: Gatsby Migration

Amino is the design system that all of the Strateos products consume. This project largely focused on streamlining the workflow for developers, while giving the users of Amino an accessible and reliable way to use Amino. You can view my final project here.

Problem: The documentation site, a separate codebase from Amino, is finnicky to update. Given how quickly Amino was expanding to supply molecular drawers/renders, along with other components, documentation was an overwhelming task for the developer.

Solution: I used Gatsby, a static site generator, to build the app. Gatsby is useful since it can query data to transform into templates. I used GraphQL to transform component data into a usage page, an interactive component demo, and a props table. I added a markdown feature in the Amino to function as a place where additional comments on component usage could be queried. There is a filter such that only components with markdown pages can be viewed on the website, because this means that the design process for these components have been deeply thought through. I integrated this into the documentation for releasing new versions of Amino, such that the website will be rebuilt and deployed whenever large feature changes are made to Amino.
